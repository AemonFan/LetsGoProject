系统概述

OVERVIEW

狼人专精系统（VocationMastery）是狼人杀游戏中的身份（职业）专精管理系统，负责展示玩家拥有的所有狼人杀身份、各身份的战绩数据、技巧说明、专精奖励领取、身份排行等功能。


整个系统基于 Lua + UE4/UMG 构建，采用 MVC + 事件驱动 架构，通过配置表驱动数据，通过服务端协议获取玩家战绩和奖励状态。

核心功能点总结

FEATURES
1

身份展示与分类

  • 按阵营分类：全部 / 平民(Prey) / 狼人(Hunter) / 中立(Neutral)
  • 支持过滤器（VocationFilter），可筛选出符合条件的身份
  • 身份卡片显示：阵营颜色背景、装饰等级（1~7级不同外观）、"新"标签、红点
2

身份解锁体系

  • 默认解锁：配置表 Preparations_DefaultUnLock == 1
  • 分数解锁：Score > 0 表示有战绩即解锁
  • 道具解锁：通过背包中的 Identity_ItemUnlock 道具判断
  • 共享身份：来自珍宝系统，有到期时间
  • 试用身份：有过期时间但非共享来源的道具
3

身份详情（三页签）

  • 基本信息：身份名称、阵营、技能描述、战绩（总场次/胜场/MVP）、分数等级、专精排名
  • 技巧进阶：通过配置表加载身份对应的技巧列表
  • 专精奖励：分数进度条 + 奖励道具展示，支持领取操作
4

专精奖励系统

  • 服务端协议：C2S_Msg 获取状态，RewardRecv 领取
  • 三种状态：-1(未解锁) / 0(可领取) / 1(已领取)
  • 进度条分段渲染（首段/中段/末段布局不同，带当前分数指示器）
  • 特殊道具处理：身份装饰道具领取后弹奖励弹窗
5

红点系统

  • 分层红点：VocationMastery侧边栏 → Tab分类 → 单个身份卡片
  • "新"标记：通过本地缓存（ClientCacheData）记录已查看身份ID
  • 奖励红点：服务端返回 red_dot 字段，领取后减1
  • 红点样式：两种（Dot 圆点 / NewMatchType "新"字样）
6

排序策略

-- 优先级排序： -- 1. 已有分数解锁 (priority=1) -- 2. 道具解锁/共享/试用 (priority=2) -- 3. 未解锁 (priority=3) -- -- 同优先级 → 分数降序 → 配置排序值升序
7

左右切换身份

  • 在详情弹窗中支持左右按钮切换前后身份
  • 通过事件 ON_NR3E3_VOCATIONMAIN_CHANGEINFO 通知主列表
8

实战练习 & 设备适配

  • 从详情页直接发起练习匹配
  • 折叠屏/特定机型：GRL-AL10、2308CPXD0C 等调整缩放比例（0.86x）
  • 胶囊区域适配：微信小游戏环境下调整Tab位置偏移

优缺点分析

EVALUATION

✅ 优点

  • 分层清晰：Model/View 职责明确，数据层与展示层解耦
  • 事件驱动解耦：各子模块通过事件通信，降低了直接依赖
  • 批量创建优化：CreateVocationItem 中采用分批帧创建（AddFrame），避免一帧创建大量Widget导致卡顿
  • 布局预计算：PreCalculateLayout 和 ApplyGridLayout 分离布局参数计算和应用，职责清晰
  • 排序策略封装：排序逻辑拆分为 GetPriority + GetSortFunction，可读性好
  • 设备兼容处理：针对折叠屏和特殊机型有专门的缩放和偏移适配
  • 红点系统完备：多层级红点（页签→分类→单项），且支持本地缓存持久化

⚠️ 缺点

  • PreparationsModel 过于庞大（3565行）：承载了大厅、匹配、活动、回归等多种职责，违反了单一职责原则
  • 硬编码较多：设备名称、特殊职业ID（45/46/47）、布局像素值（355.0, 236.0等）散布在代码中
  • 进度条逻辑复杂度高：ProgressItem:InitUI 中的分段计算逻辑嵌套较深，缺少注释说明公式含义
  • 红点逻辑冗余：红点的"新"标记逻辑在多个文件中都有部分实现，逻辑分散且有重复
  • 内存清理风格不统一：OnClose 中有的手动置nil，有的依赖框架清理，缺乏统一规范
  • Magic Number：如 278.0 * progress + 78 这类数字缺少常量定义
  • 注释不足：核心算法（排序优先级、进度条分段公式）和业务规则缺少文档性注释
  • goto continue 用法：在某些循环中使用降低了可读性
