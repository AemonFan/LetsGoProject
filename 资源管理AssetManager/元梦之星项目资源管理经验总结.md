	Chunk与.pak的关系：	一个 Chunk 可以包含多个 .pak 文件，它们是一对多的关系。（可参考：struct FChunk 与 struct FPakFile）
		两者的职责区别:
		概念	       类型	                职责
		Chunk	   逻辑单元（FChunk）	    以 ChunkId 为标识，是游戏逻辑层面的"功能包"，例如某个玩法、某个关卡的所有资源集合
		.pak 文件   物理文件（FPakFile）	实际存储在磁盘/CDN 上的压缩文件，是下载和挂载的最小物理单元
		
		为什么要这样设计？从 FPakFileEntry 的定义可以看出：
			struct FPakFileEntry
			{
				FString FileName;       // pak 文件名（唯一）
				uint64 FileSize = 0;    // 文件大小（字节）
				FString FileVersion;    // 版本/SHA1 哈希
				int32 ChunkId = -1;     // 归属的 ChunkId ← 多个 PakFile 可以有相同的 ChunkId
				FString RelativeUrl;    // CDN 上的相对路径
			};
		
		这样设计的好处是：
			控制单个 .pak 文件大小：单个 .pak 文件过大会导致下载失败后需要重新下载整个文件，拆分成多个小 .pak 可以支持断点续传和并发下载
			灵活的热更粒度：某个 Chunk 中只有部分资源更新时，只需要替换对应的 .pak 文件，而不是整个 Chunk 重新下载
			并发下载：同一个 Chunk 的多个 .pak 文件可以同时下载（TargetDownloadsInFlight 控制并发数），加快整体下载速度
			挂载顺序控制：MountChunkInternal 会按顺序挂载同一 Chunk 内的所有 .pak 文件，保证资源覆盖关系正确
		
		操作粒度的对应关系
			下载（Download）：以 .pak 文件 为单位（DownloadPakFileInternal）
			挂载（Mount）：以 Chunk 为单位（MountChunk），但内部会依次挂载该 Chunk 下的所有 .pak 文件
			卸载（Unmount）：以 .pak 文件 为单位（UnmountPakFile）
			游戏逻辑感知：以 Chunk 为单位（GetChunkStatus、OnChunkMounted）

	资源分包：
		元梦之星项目项目庞大，玩法多样，每个玩法可分为多个资源包，每个资源包可单独下载（DownloadChunks），减少内存占用；且支持按需加载（MountChunks）/卸载，避免一次性加载过多资源，降低内存峰值。
		狼人玩法只需下载基础包（小包）资源即可开始游戏，商业化资源等在资源包（大包）中，需要额外下载才能使用。
		
	AssetNameMapping资源映射：元梦之星项目AssetNameMapping资源映射.md
		资源名称为key，映射资源在引擎中的路径，映射文件可热更，进入某个玩法Mount加载玩法对应的AssetNameMapping，退出玩法后UnMount卸载AssetNameMapping,减少AssetNameMapping映射文件本身占用的内存空间，同时起到空间隔离的作用，且能够减少资源查询区间，保持资源查询性能稳定。

	项目中资源加载失败的原因总结：
		1、资源本身不存在导致加载失败
			1）资源未上传至引擎
			2）cdn资源未上传至cdn 或 cdn资源路径不正确
		2、资源引用丢失
			1）UI蓝图中直接引用资源，资源路径改动可能导致资源丢失，需要重新引用并编译UI蓝图
			2）UI蓝图中引用了其他目录的资源，在手机端分包后，其他目录的资源如果没有加载，则当前引用会丢失
		3、资源格式不正确导致资源加载失败
			1）资源过大（如资源最大为5M、图片大小不超过1920*1080）
			2）资格类型不支持（支持动态加载的UI资源类型有：astc\png\jpg\map）
			3）资源格式不正确（如astc压缩格式的图片宽度和高度必须是块大小的整数倍，图片下载失败案例：astc块大小为8*8像素，图片尺寸为408x420像素，408 ÷ 8 = 51（整数，宽度符合要求），420 ÷ 8 = 52.5（不是整数，高度不符合要求））
		4）资源动态加载AssetNameMapping映射文件相关原因
			1）动态加载的资源必须导出AssetNameMapping文件，否则在加载时无法从资源映射表中找到资源的全路径导致加载失败
			2）部分类型的资源需要在AssetDirectorySetTable_NR3E表中添加导出路径，才会导出到AssetNameMapping_NR3E文件中，如(AssetFilter=/Script/Engine.Texture2D,AssetDirectory="/Game/Feature/NR3E")
			3）AssetNameMapping按玩法需求加载，退出玩法后对应的AssetNameMapping文件会卸载，如果在此玩法外引用了玩法的资源会导致加载失败
			（案例：如主玩法需要在祈愿活动中预览狼人的局内道具：会议称号，会议称号的预览中部分UI资源是在狼人分包中的，局外只有在狼人备战场景才会加载AssetNameMapping_NR3E.txt文件，退出后卸载，会导致祈愿预览场景无法找到狼人的UI资源而报错。）
