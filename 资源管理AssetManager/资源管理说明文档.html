🏗️ 整体架构概览

LetsGo 的资源管理体系由三个层次协同工作，形成完整的资源生命周期管理链路。

逻辑层
📄 AssetNameMapping 文件
短名 → 长路径 映射表，支持按玩法 Mount/UnMount
🎮 UMoeAssetManager
资源加载/卸载/查询封装，内置性能分析与失败上报
▼ 短名查询 → FSoftObjectPath ▼
物理层
📦 Chunk（逻辑分包）
以 ChunkId 为标识的功能包，如某个玩法的所有资源集合
💾 .pak 文件
磁盘/CDN 上的压缩文件，下载和挂载的最小物理单元
▼ Mount 后资源可访问 ▼
引擎层
⚙️ FStreamableManager
UE4 流式加载核心，管理 UObject 资源实例的生命周期
🌐 FChunkDownloader
分包下载/挂载管理器，支持并发下载与断点续传
💡
核心设计思路： 绕过 UAssetManager 重量级的 PrimaryAsset 体系，直接封装 FStreamableManager， 自建短名映射系统，配合 Chunk 分包机制，实现资源的按需加载、按玩法隔离、支持热更的完整资源管理方案。

⚖️ UMoeAssetManager vs UAssetManager

元梦之星为何不直接使用 UE4 原生的 UAssetManager？两者在设计哲学上有根本差异。

维度 UAssetManager（UE4 原生） UMoeAssetManager（元梦定制）
定位 通用引擎框架，兼顾编辑器 + 运行时 专为手游运行时性能优化
资产寻址 FPrimaryAssetId（类型+名称） 短名字符串（如 "hero_mesh"）
资产注册 依赖 AssetRegistry 全量扫描磁盘 读取轻量级映射文件（txt / FlatBuffers）
实例模式 全局单例 非单例，由游戏框架管理
热更支持 不支持运行时动态替换路径 映射文件可随热更包下发动态替换
启动速度 全量扫描拖慢手游启动 流式读取映射文件，启动极快
内存占用 AssetRegistry 缓存可达数十 MB 4KB 分块流式读取，峰值极低

UMoeAssetManager 的五大核心优势

① 短名映射系统 — 业务与路径解耦

业务代码只需使用短名，资产移动/重命名只需更新映射文件，无需改动业务代码。

C++ // 业务代码永远不需要知道完整路径
LoadSyncAssetObject<USkeletalMesh>("hero_mesh");

② FlatBuffers + MMap 极致性能

提供三套查询后端，按平台自动选择最优方案：

模式适用场景
普通 TMap通用场景
Ansi StringView减少 Unicode 内存开销
FlatBuffers MMapiOS/Mac，零拷贝，OS 管理内存

③ 流式读取 — 内存友好

4KB 分块流式读取映射文件，避免大文件一次性加载到内存，将峰值内存压到最低。

C++ constexpr int32 BufferSize = 4 * 1024;
TArray<uint8> Buffer;
Buffer.SetNumUninitialized(BufferSize);
// 逐块读取并解析...

④ 内置性能分析埋点

所有同步/异步加载接口都内置了性能采集代理，可实时监控每个资产的加载耗时。

C++ // 每次加载都会广播性能数据
if(UMoeAssetLoadAnalysisSubsystem::
  OnMoeProfilerCollectAssetLoading.IsBound())
  OnMoeProfilerCollectAssetLoading.Broadcast(
    StartTime, StartSeconds, {ObjectPath}, false);

⑤ 线程安全的加载失败上报

配合 Pak 错误事件监听，精准定位线上资产缺失问题。

C++ static TArray<FString> FailedAssetNames;
static FCriticalSection FailedAssetNamesCriticalSection;

⑥ UI 资源专项支持

针对 UI 系统提供颜色表、字体表、DataTable 快捷访问接口，并带有缓存，避免重复加载。

C++ FLinearColor GetUserWidgetColor(const FString& ColorName);
FSlateFontInfo GetUserWidgetFont(const FString& FontName);
UDataTable* GetUserWidgetTable(const FString& Name);

🗺️ AssetNameMapping 短名映射系统

资源名称作为 key，映射资源在引擎中的完整路径。映射文件支持热更，是整个资源管理体系的核心寻址机制。

映射文件工作原理

AssetNameMapping 本质是一张短名 → 长路径的字符串字典。每条记录将业务层使用的短名映射到引擎中的完整资源路径。

映射示例 // AssetNameMapping 文件内容（txt 格式）
hero_mesh → /Game/Characters/Hero/SK_Hero.SK_Hero
hero_anim → /Game/Characters/Hero/Anim/ABP_Hero.ABP_Hero_C
lobby_bg → /Game/UI/Lobby/T_LobbyBG.T_LobbyBG
  • ✅ 支持热更 — 映射文件可随热更包下发，实现路径动态替换
  • ✅ 支持多包合并 — MergeAssetNameMappingByPathAndKey 合并多个映射文件到同一 Key
  • ✅ 按玩法隔离 — 每个玩法独立映射文件，Mount/UnMount 管理
  • ✅ 预留 80000 条目容量 — 满足大型项目资产规模
🎯
关键优势：业务代码与资产路径完全解耦，资产移动/重命名只需更新映射文件，不需要修改任何业务代码。 策划/程序习惯用短名配置资产，大幅降低协作成本。

🔄 按玩法 Mount / UnMount 机制

进入某个玩法时加载对应的 AssetNameMapping 映射文件，退出玩法后卸载，实现内存按需管理与命名空间隔离。

工作流程

1. 进入玩法 → InitAssetNameMappingByPath()
加载该玩法的映射文件，写入 TMap<短名 → 路径>，此时仅占用字符串内存，资源未加载
2. 加载资源 → LoadSyncAssetObject("hero_mesh")
通过映射表查询短名 → 得到 FSoftObjectPath → FStreamableManager 加载实际 UObject
3. 退出玩法 → 先卸载资源，再清除映射表
调用 UnloadAssetObject() 卸载已加载资源，再调用 ClearAssetNameMappingByPath() 释放映射字典内存
4. 等待 GC → 真正回收 UObject 内存
资源引用计数归零后，下次 GC 时真正回收内存

四大设计优势

🧠 映射表内存按需管理

每个玩法的 AssetNameMappingInfo 在退出时通过 TSharedPtr 引用计数归零完全释放， 包括 ShortNameToPath、AssetKVStringStore 等数据结构全部回收。

🔒 命名空间隔离

不同玩法可以存在同名短名（如都有 "hero_mesh"），各玩法的映射表是独立的 AssetNameMappingInfo 实例，互不干扰。

♻️ 热更动态替换

退出后卸载映射文件，下次进入时重新加载最新版本，天然支持资产路径的热更替换，无需重启游戏。

⚡ 查询性能稳定

GetAssetObjectPath 遍历 AssetNameMappingInfoMap，按需 Mount 保证 Map 中始终只有当前活跃玩法的映射，查询复杂度保持在较低水平。

⚠️
关键认知：卸载映射文件 ≠ 卸载资源！
ClearAssetNameMappingByPath 只释放字符串字典内存，不触发任何 UObject 资源的卸载。 资源的实际生命周期由 FStreamableManager 控制。

映射表 vs 资源 — 生命周期对比

操作映射表内存资源（UObject）内存
InitAssetNameMappingByPath ✅ 分配（加载短名→路径字典） ❌ 不变（资源未加载）
LoadSyncAssetObject / RequestAsyncLoad ❌ 不变 ✅ 资源加载进内存
ClearAssetNameMappingByPath ✅ 释放（字典内存回收） ❌ 资源仍在内存中
UnloadAssetObject ❌ 不变 ✅ 资源从 StreamableManager 释放
GC 触发 ❌ 不变 ✅ 无引用的资源被 GC 回收
🚨
正确的退出顺序：
① 先调用 UnloadAssetObject() 卸载资源（此时映射表还在，能查到路径）
② 再调用 ClearAssetNameMappingByPath() 清除映射表
③ 等待 GC 触发，真正回收 UObject 内存

若顺序颠倒（先清映射再卸资源），GetAssetObjectPath 找不到路径，资源将无法被主动卸载！

📦 Chunk 与 .pak 文件

Chunk 是逻辑分包单元，.pak 是物理文件单元，两者构成一对多的层级关系。

层级关系

结构 Chunk（逻辑单元，ChunkId = 101）
  ├── PakFile_101_001.pak  ← FPakFile（120MB）
  ├── PakFile_101_002.pak  ← FPakFile（95MB）
  └── PakFile_101_003.pak  ← FPakFile（88MB）

Pak 文件数据结构

C++ struct FPakFileEntry
{
  FString FileName;       // pak 文件名（唯一）
  uint64 FileSize = 0;   // 文件大小（字节）
  FString FileVersion;    // 版本/SHA1 哈希
  int32 ChunkId = -1;    // 归属的 ChunkId ← 多个 Pak 可以有相同的 ChunkId
  FString RelativeUrl;    // CDN 上的相对路径
};

操作粒度对应关系

操作粒度接口
下载 Download 以 .pak 文件 为单位 DownloadPakFileInternal
挂载 Mount 以 Chunk 为单位（内部依次挂载所有 .pak） MountChunkInternal
卸载 Unmount 以 .pak 文件 为单位 UnmountPakFile
游戏逻辑感知 以 Chunk 为单位 GetChunkStatus、OnChunkMounted

拆分为多个 .pak 的设计价值

🔄

断点续传

单个 .pak 下载失败只需重下该文件，不需要重新下载整个 Chunk

⚡

并发下载

同一 Chunk 的多个 .pak 可并行下载，TargetDownloadsInFlight 控制并发数

🎯

精细热更

只有部分资源更新时，只替换对应的 .pak 文件，不重下整个 Chunk

🔄 资源完整生命周期

资源分包方案：项目庞大，玩法多样，每个玩法可分为多个资源包，支持按需下载和加载。

分包策略

以狼人玩法为例，展示资源分包的实际应用：

📱 基础包（小包）

包含核心玩法资源，下载即可开始游戏。如狼人玩法的基础 UI、角色模型、核心逻辑资源等。

💎 资源包（大包）

包含商业化资源、高清材质、额外皮肤等，需要额外下载才能使用。

资源生命周期全链路

① 下载（DownloadChunks）
从 CDN 下载 .pak 文件到本地，支持并发下载、断点续传，以 .pak 文件为最小下载单位
② 挂载（MountChunks）
将 Chunk 对应的 .pak 文件挂载到引擎的文件系统中，使其中的资源可以被访问，按顺序挂载保证覆盖关系正确
③ 加载映射表（InitAssetNameMappingByPath）
加载对应玩法的 AssetNameMapping 文件，建立短名到长路径的映射关系
④ 加载资源（LoadSyncAssetObject / RequestAsyncLoad）
通过短名查找到完整路径，经由 FStreamableManager 加载 UObject 资源到内存
⑤ 使用资源
游戏运行中正常使用已加载的资源
⑥ 卸载资源（UnloadAssetObject）→ 清除映射（ClearAssetNameMappingByPath）
退出玩法时，先通过映射表找到资源路径并卸载，再清除映射表本身
⑦ GC 回收
引用计数归零后的 UObject 在下次垃圾回收时真正释放内存

🔍 资源加载失败常见原因与排查

项目中总结的资源加载失败典型场景及解决方案，帮助快速定位和解决问题。

❌ 原因一：资源本身不存在

  • 资源未上传至引擎
  • CDN 资源未上传至 CDN 或 CDN 资源路径不正确

🔗 原因二：资源引用丢失

  • UI 蓝图中直接引用资源，资源路径改动可能导致资源丢失，需要重新引用并编译 UI 蓝图
  • UI 蓝图中引用了其他目录的资源，在手机端分包后，其他目录的资源如果没有加载，则当前引用会丢失

📐 原因三：资源格式不正确

  • 资源过大 — 如资源最大为 5MB、图片大小不超过 1920×1080
  • 资源类型不支持 — 支持动态加载的 UI 资源类型：astc / png / jpg / map
  • ASTC 格式约束 — astc 压缩格式的图片宽度和高度必须是块大小的整数倍
📋
真实案例：astc 块大小为 8×8 像素，图片尺寸为 408×420 像素。 408 ÷ 8 = 51（✅ 整数）, 420 ÷ 8 = 52.5（❌ 不是整数）→ 高度不符合要求，图片下载失败。

🗺️ 原因四：AssetNameMapping 映射文件相关

  • 动态加载的资源必须导出 AssetNameMapping 文件，否则无法从映射表中找到全路径
  • 部分类型的资源需要在 AssetDirectorySetTable_NR3E 表中添加导出路径
    如：AssetFilter=/Script/Engine.Texture2D, AssetDirectory="/Game/Feature/NR3E"
  • AssetNameMapping 按玩法需求加载，退出玩法后对应文件会卸载，在此玩法外引用玩法资源会导致加载失败
🎮
真实案例：主玩法需要在祈愿活动中预览狼人的局内道具"会议称号"。 会议称号的预览中部分 UI 资源在狼人分包中，局外只有在狼人备战场景才会加载 AssetNameMapping_NR3E.txt 文件， 退出后卸载，导致祈愿预览场景无法找到狼人的 UI 资源而报错。
