📦 Chunk 与 .pak 文件机制

Chunk 是逻辑分包单元，.pak 是物理文件单元。两者构成一对多的层级关系，支撑资源的按需下载、挂载和热更。

断点续传 并发下载 异步挂载 状态持久化
来源：元梦之星项目资源管理经验总结.md、.pak 文件在游戏中的解析与加载流程.md、.pak 资源挂载的完整触发流程.md、Chunk 下载状态的标记与持久化机制.md

🗂️ 一对多层级关系

一个 Chunk 可以包含多个 .pak 文件。

结构 Chunk（逻辑单元，ChunkId = 101） ├── PakFile_101_001.pak ← FPakFile（120MB） ├── PakFile_101_002.pak ← FPakFile（95MB） └── PakFile_101_003.pak ← FPakFile（88MB）
概念类型职责
Chunk逻辑单元（FChunk）以 ChunkId 为标识，是游戏逻辑层面的"功能包"，如某个玩法的所有资源集合
.pak 文件物理文件（FPakFile）实际存储在磁盘/CDN 上的压缩文件，是下载和挂载的最小物理单元

Pak 文件数据结构

C++ struct FPakFileEntry { FString FileName; // pak 文件名（唯一） uint64 FileSize = 0; // 文件大小（字节） FString FileVersion; // 版本/SHA1 哈希 int32 ChunkId = -1; // 归属的 ChunkId ← 多个 Pak 可以有相同 ChunkId FString RelativeUrl; // CDN 上的相对路径 };

操作粒度对应关系

操作粒度接口
下载 Download以 .pak 文件 为单位DownloadPakFileInternal
挂载 Mount以 Chunk 为单位（内部依次挂载所有 .pak）MountChunkInternal
卸载 Unmount以 .pak 文件 为单位UnmountPakFile
游戏逻辑感知以 Chunk 为单位GetChunkStatus、OnChunkMounted

拆分为多个 .pak 的设计价值

🔄

断点续传

单个 .pak 下载失败只需重下该文件，不需要重新下载整个 Chunk

⚡

并发下载

同一 Chunk 的多个 .pak 可并行下载，TargetDownloadsInFlight 控制并发数

🎯

精细热更

只有部分资源更新时，只替换对应的 .pak 文件，不重下整个 Chunk

📋 .pak 文件物理结构

.pak 文件从末尾开始读取 Footer，再定位到主索引和二级索引。

物理结构 ┌─────────────────────────────────────────────────────┐ │ 资源数据区（Payload） │ │ [File1 Data][File2 Data][File3 Data]... │ │ （可压缩：Zlib/Gzip/Oodle，可加密：AES） │ ├─────────────────────────────────────────────────────┤ │ 主索引（Primary Index） │ │ MountPoint + NumEntries + PathHashSeed │ │ + EncodedPakEntries（每个文件的 Offset/Size/Hash） │ ├─────────────────────────────────────────────────────┤ │ 路径哈希索引（PathHashIndex） │ │ FName Hash → FPakEntryLocation（快速查找） │ ├─────────────────────────────────────────────────────┤ │ 完整目录索引（FullDirectoryIndex） │ │ TMap<目录路径, TMap<文件名, FPakEntryLocation>> │ ├─────────────────────────────────────────────────────┤ │ 文件尾（Footer / FPakInfo） │ │ Magic(4B) + Version(4B) + IndexOffset(8B) │ │ + IndexSize(8B) + IndexHash(20B) + EncryptedFlag │ │ + EncryptionKeyGuid + CompressionMethods │ └─────────────────────────────────────────────────────┘ ↑ 从文件末尾开始读取
索引类型数据结构内存占用用途
PathHashIndexTMap<uint64 Hash, FPakEntryLocation>小（仅哈希值）运行时快速查找（默认）
FullDirectoryIndexTMap<目录, TMap<文件名, Location>>大（完整路径字符串）目录遍历、编辑器

🔌 .pak 资源挂载完整触发流程

整个挂载过程分为 5 个阶段，跨越 ChunkDownloader → FCoreDelegates → FPakPlatformFile 三个模块。

sequenceDiagram
    participant Game as 游戏逻辑层
    participant CD as FChunkDownloader
    participant MT as FPakMountWork（异步线程）
    participant PPF as FPakPlatformFile
    participant PakFile as FPakFile

    Game->>CD: MountChunk(ChunkId, Callback)
    CD->>CD: 检查 Chunk.bIsMounted?
    alt 已挂载
        CD-->>Game: ExecuteNextTick(Callback, true)
    else 未挂载 且 已缓存
        CD->>MT: 创建 FMountTask → StartBackgroundTask()
        MT->>MT: 遍历 PakFiles，计算 PakReadOrder
        MT->>PPF: FCoreDelegates::OnMountPak.Execute()
        PPF->>PakFile: new FPakFile → 解析 Footer+Index
        PPF->>PPF: PakFiles.Add → 加入已挂载列表
        MT-->>CD: CompleteMountTask() → bIsMounted=true
        CD-->>Game: Callback(true)
    else 未缓存
        CD->>CD: DownloadChunkInternal → 下载完成后重新 MountChunk
    end

阶段一：MountChunk — 入口判断（主线程）

C++ void FChunkDownloader::MountChunk(int32 ChunkId, const FCallback& Callback) { TSharedRef<FChunk>* ChunkPtr = Chunks.Find(ChunkId); if (Chunk.bIsMounted) { ExecuteNextTick(Callback, true); // 已挂载，直接回调 return; } MountChunkInternal(Chunk, Callback); // 进入挂载流程 }

阶段二：MountChunkInternal — 决策分支

路径A：所有 pak 已缓存 → 直接启动异步挂载任务
路径B：有 pak 未缓存 → 先下载，下载完成后再触发挂载

C++ // 创建异步任务 Chunk.MountTask = new FMountTask(); FPakMountWork& MountWork = Chunk.MountTask->GetTask(); MountWork.CacheFolder = CacheFolder; for (const TSharedRef<FPakFile>& PakFile : Chunk.PakFiles) { if (!PakFile->bIsMounted) MountWork.PakFiles.Add(PakFile); } Chunk.MountTask->StartBackgroundTask(); // 启动异步线程

阶段三：FPakMountWork::DoWork — 实际挂载（异步线程）

C++ void DoWork() { uint32 PakReadOrder = PakFiles.Num(); for (const TSharedRef<FPakFile>& PakFile : PakFiles) { FString FullPathOnDisk = CacheFolder / PakFile->Entry.FileName; // 通过委托触发实际挂载 bool bMountOk = FCoreDelegates::OnMountPak.Execute( FullPathOnDisk, PakReadOrder, nullptr); if (bMountOk) --PakReadOrder; } }
💡
关键设计：PakReadOrder 从 PakFiles.Num() 开始递减，确保 Chunk 内的 pak 文件按顺序优先级挂载。

阶段四：FPakPlatformFile::Mount — 底层挂载

C++ bool FPakPlatformFile::Mount(const TCHAR* InPakFilename, uint32 PakOrder, ...) { // 1. 打开文件句柄 TSharedPtr<IFileHandle> PakHandle = LowerLevel->OpenRead(InPakFilename); // 2. 解析 pak 文件（读取 Footer + Index） FPakFileThreadSafePtr Pak = MakeShared<FPakFile>(LowerLevel, InPakFilename, ...); if (Pak->IsValid()) { // 3. 加入已挂载列表（线程安全） FScopeLock ScopedLock(&PakListCritical); PakFiles.Add(Entry); PakFiles.StableSort(); // 按优先级排序 // 4. 广播挂载完成通知 FCoreDelegates::OnPakFileMounted2.Broadcast(*Pak); } }

💾 Chunk 下载状态的标记与持久化

核心设计：内存状态是运行时的，持久化状态靠磁盘上的 LocalManifest.txt 文件。

运行时内存状态

C++ struct FPakFile { FPakFileEntry Entry; // 元数据（FileName, FileSize, ChunkId...） bool bIsCached = false; // ✅ pak 是否已完整下载到磁盘 bool bIsMounted = false; // ✅ pak 是否已挂载到内存文件系统 bool bIsEmbedded = false; // 是否是随包内置的 pak uint64 SizeOnDisk = 0; // 磁盘上的实际文件大小 };

LocalManifest.txt 持久化文件

存储路径：FPaths::ProjectPersistentDownloadDir() / "PakCache/LocalManifest.txt"

文件格式（Tab 分隔） $NUM_ENTRIES = 3 GameplayA_001.pak 102400 v1.4.22-r23928 -1 / GameplayA_002.pak 204800 v1.4.22-r23928 -1 / GameplayB_001.pak 512000 v1.4.22-r23928 -1 /
sequenceDiagram
    participant Game as 游戏启动
    participant CD as FChunkDownloader::Initialize()
    participant Disk as 磁盘文件系统
    participant Mem as 内存 PakFiles Map

    Game->>CD: Initialize()
    CD->>Disk: 读取 PakCache/LocalManifest.txt
    CD->>Disk: FileManager.FileSize(CacheFolder / Entry.FileName)
    alt 文件存在 且 大小 == Entry.FileSize
        CD->>Mem: bIsCached = true ✅
    else 文件存在 但 大小 < Entry.FileSize
        CD->>Mem: SizeOnDisk = 实际大小（部分下载）
    else 文件不存在
        CD->>Mem: 从 LocalManifest 移除
    end
    CD->>Disk: SaveLocalManifest() 清理无效条目

📊 Chunk 状态生命周期

GetChunkStatus 通过遍历 Chunk 下的 PakFiles 计算整体状态。

graph TD
    A["GetChunkStatus(ChunkId)"] --> B{"Chunk.bIsMounted?"}
    B -->|Yes| C["Mounted（已挂载到内存）"]
    B -->|No| D["统计 NumCached / NumDownloading"]
    D --> E{"NumCached >= NumPaks?"}
    E -->|Yes| F["Cached（全部下载完成，未挂载）"]
    E -->|No| G{"NumCached + NumDownloading >= NumPaks?"}
    G -->|Yes| H["Downloading（下载中）"]
    G -->|No| I{"NumCached + NumDownloading > 0?"}
    I -->|Yes| J["Partial（部分下载，已暂停）"]
    I -->|No| K["Remote（未开始下载）"]

状态生命周期总结

状态流转 首次下载： Remote → (DownloadChunk) → Downloading → (下载完成) → Cached → (MountChunk) → Mounted 游戏重启后： 磁盘有完整 pak → Initialize() → bIsCached=true → Cached（无需重新下载） 磁盘有部分 pak → Initialize() → SizeOnDisk>0, bIsCached=false → Partial（支持断点续传） 磁盘无 pak 文件 → LocalManifest 条目被清除 → Remote（需要重新下载）
💡
关键结论：bIsMounted 是纯内存状态，游戏重启后永远从 false 开始，必须重新调用 MountChunk。而 bIsCached 通过 LocalManifest.txt + 磁盘文件大小校验来恢复，这就是已下载的玩法资源在重启后不需要重新下载的原因。

🎮 元梦之星的定制扩展

在标准 UE4 pak 格式上做了安全性增强。

定制点说明
PakFile_Version_ZplanCustom (v12)在 Footer 中增加了 FZplanEngineVersion，用于引擎版本兼容性校验，防止旧版本 pak 被新引擎错误加载
PakFile_Version_ZplanEncrypted (v13)在 FPakEntry 的 Offset 字段位置插入随机数混淆（PakEntry_Magic = 0x709665BA），增加逆向难度
OutErrorCode 参数在 Mount/LoadIndex 全链路增加了精细化错误码（EPME_FailurePakIntegrityError 等），方便线上问题定位

📝 关键设计总结

挂载流程中的核心设计决策。

设计点说明
委托解耦ChunkDownloader 通过 FCoreDelegates::OnMountPak 委托调用挂载，不直接依赖 FPakPlatformFile
异步挂载FPakMountWork 在后台线程执行，避免阻塞游戏主线程
Ticker 轮询主线程通过每帧 Ticker 检测异步任务完成，保证回调在主线程执行
顺序优先级PakReadOrder 递减确保同一 Chunk 内 pak 文件按正确顺序覆盖
加密延迟挂载若加密 Key 未就绪，pak 进入 PendingEncryptedPakFiles 队列，Key 注册后自动补挂
下载→挂载串联未缓存时先下载，下载完成回调中再次调用 MountChunk，形成完整链路
