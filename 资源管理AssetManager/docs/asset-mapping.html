🗺️ AssetNameMapping 短名映射系统

资源名称作为 key，映射资源在引擎中的完整路径。映射文件支持热更，是整个资源管理体系的核心寻址机制。

支持热更 按玩法隔离 多包合并
来源：元梦之星项目AssetNameMapping资源映射.md、元梦之星项目资源管理经验总结.md

📖 映射文件工作原理

AssetNameMapping 本质是一张 短名 → 长路径 的字符串字典。每条记录将业务层使用的短名映射到引擎中的完整资源路径。

映射示例 // AssetNameMapping 文件内容（txt 格式） hero_mesh → /Game/Characters/Hero/SK_Hero.SK_Hero hero_anim → /Game/Characters/Hero/Anim/ABP_Hero.ABP_Hero_C lobby_bg → /Game/UI/Lobby/T_LobbyBG.T_LobbyBG
  • ✅ 支持热更 — 映射文件可随热更包下发，实现路径动态替换
  • ✅ 支持多包合并 — MergeAssetNameMappingByPathAndKey 合并多个映射文件到同一 Key
  • ✅ 按玩法隔离 — 每个玩法独立映射文件，Mount/UnMount 管理
  • ✅ 预留 80000 条目容量 — 满足大型项目资产规模
🎯
关键优势：业务代码与资产路径完全解耦，资产移动/重命名只需更新映射文件，不需要修改任何业务代码。策划/程序习惯用短名配置资产，大幅降低协作成本。

🔗 映射表与资源的关系

映射表仅是索引/目录，不持有任何 UObject 资源引用，也不参与 UE 的 GC 系统。

🔄 按玩法 Mount / UnMount 机制

进入某个玩法时加载对应的 AssetNameMapping 映射文件，退出玩法后卸载。实现内存按需管理与命名空间隔离。

sequenceDiagram
    participant Game as 游戏逻辑
    participant Mgr as UMoeAssetManager
    participant Map as AssetNameMappingInfoMap
    participant SM as FStreamableManager

    Game->>Mgr: 进入玩法 → InitAssetNameMappingByPath()
    Mgr->>Map: 加载映射文件，写入 TMap（短名→路径）
    Note over Map: 仅占用字符串内存，资源未加载

    Game->>Mgr: 加载资源 → LoadSyncAssetObject("hero_mesh")
    Mgr->>Map: 查询短名 → 得到 FSoftObjectPath
    Mgr->>SM: RequestSyncLoad(FSoftObjectPath)
    SM-->>Game: 返回 UObject*

    Game->>Mgr: 退出玩法
    Mgr->>SM: Unload(FSoftObjectPath) — 先卸载资源！
    Mgr->>Map: 移除该玩法的映射条目 — 再清除映射！

工作流程详解

1. 进入玩法 → InitAssetNameMappingByPath()
加载该玩法的映射文件，写入 TMap<短名 → 路径>，此时仅占用字符串内存，资源未加载
2. 加载资源 → LoadSyncAssetObject("hero_mesh")
通过映射表查询短名 → 得到 FSoftObjectPath → FStreamableManager 加载实际 UObject
3. 退出玩法 → 先卸载资源，再清除映射表
调用 UnloadAssetObject() 卸载已加载资源，再调用 ClearAssetNameMappingByPath() 释放映射字典内存
4. 等待 GC → 真正回收 UObject 内存
资源引用计数归零后，下次 GC 时真正回收内存

Mount/UnMount 的四大设计优势

🧠 映射表内存按需管理

每个玩法的 AssetNameMappingInfo（预留 80000 条目）在退出时通过 TSharedPtr 引用计数归零完全释放，包括 ShortNameToPath、AssetKVStringStore 等数据结构全部回收。

C++ // Init 时记录了加载前后的内存差值 uint64 UsedMemoryBefore = MemoryStatsBefore.UsedPhysical; // ... 加载映射文件 ... uint64 UsedMemoryAfter = MemoryStatsAfter.UsedPhysical; MOE_LOG(LogMoeAssetManager, Log, TEXT("InitAssetNameMappingByPath Diff %lld"), UsedMemoryAfter - UsedMemoryBefore);

🔒 命名空间隔离

不同玩法可以存在同名短名（如都有 "hero_mesh"），各玩法的映射表是独立的 AssetNameMappingInfo 实例，互不干扰。退出玩法后清除该玩法的映射，不影响其他玩法。

♻️ 热更动态替换

映射文件可以随热更包下发，进入玩法时加载最新的映射文件，退出后卸载。下次进入时重新加载，天然支持资产路径的热更替换，不需要重启游戏。

⚡ 查询性能稳定

GetAssetObjectPath 遍历 AssetNameMappingInfoMap 的所有条目。按需 Mount/UnMount 保证 Map 中始终只有当前活跃玩法的映射，查询复杂度保持在较低水平。

C++ for(auto& AssetNameMappingInfo : AssetNameMappingInfoMap) { const FString* FoundPath = AssetNameMappingInfo.Value->ShortNameToPath.Find(LowerShortName); ... }

⚠️ 关键认知：卸载映射文件 ≠ 卸载资源

这是这套设计中最关键的一个区分点。

⚠️
卸载映射文件 ≠ 卸载资源！
ClearAssetNameMappingByPath 只释放字符串字典内存，不触发任何 UObject 资源的卸载。资源的实际生命周期由 FStreamableManager 控制。

映射表 vs 资源 — 生命周期对比

操作映射表内存资源（UObject）内存
InitAssetNameMappingByPath ✅ 分配（加载短名→路径字典） ❌ 不变（资源未加载）
LoadSyncAssetObject / RequestAsyncLoad ❌ 不变 ✅ 资源加载进内存
ClearAssetNameMappingByPath ✅ 释放（字典内存回收） ❌ 资源仍在内存中
UnloadAssetObject ❌ 不变 ✅ 资源从 StreamableManager 释放
GC 触发 ❌ 不变 ✅ 无引用的资源被 GC 回收

UnloadAssetObject 实际卸载逻辑

C++ // UnloadAssetObject 才是真正触发资源卸载的接口 void UMoeAssetManager::UnloadAssetObject(const FString& AssetNameKey) { // 先通过短名查到长路径 const FSoftObjectPath* AssetPath = GetAssetObjectPath(AssetNameKey); if (AssetPath != nullptr && AssetPath->IsValid()) { TArray<TSharedRef<FStreamableHandle>> HandleList; if (StreamableManager->GetActiveHandles(*AssetPath, HandleList)) { StreamableManager->Unload(*AssetPath); // 这才是真正卸载资源 } } }
🚨
正确的退出顺序：
① 先调用 UnloadAssetObject() 卸载资源（此时映射表还在，能查到路径）
② 再调用 ClearAssetNameMappingByPath() 清除映射表
③ 等待 GC 触发，真正回收 UObject 内存

若顺序颠倒（先清映射再卸资源），GetAssetObjectPath 找不到路径，资源将无法被主动卸载！

⏱️ 资源实际卸载时机

退出玩法时，资源真正被释放需要满足以下条件之一。

主动调用 UnloadAssetObject
在玩法退出逻辑中逐一卸载已加载的资源
FStreamableHandle 超出作用域
如果加载时使用的 Handle 被释放（bManageActiveHandle = false 且 Handle 析构），StreamableManager 会释放对资源的引用
等待 GC
资源引用计数归零后，下次 GC 时才真正回收内存
